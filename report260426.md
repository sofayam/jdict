# Session report — 2026-04-26

## 1. Kanjis tab in the wiki index

Added a new **Kanjis** tab to the wiki index page (`/wiki#kanjis`) showing a grid of the 2500 most frequent kanji (ordered by the `freq` field in KANJIDIC2).

### Colour coding

| Colour | Meaning |
|--------|---------|
| Blue   | Kanji appears in at least one wiki headword (slug) |
| Green  | Kanji appears in wiki notes/body but not in any headword |
| Grey   | Kanji not found anywhere in the wiki |

Blue takes priority over green. The grid is lazy-loaded the first time the tab is activated (single `GET /api/wiki/kanji-coverage` fetch).

### Kanji detail page — `/wiki/kanji/:char`

Clicking any non-grey kanji navigates to a detail page listing:
- On/kun readings and English meanings from KANJIDIC2
- Grade, frequency rank
- **In headwords** — wiki word pages whose slug contains this kanji
- **In notes only** — wiki word pages whose notes contain this kanji (but not the slug)

### New backend

**`wiki.js`**
- `getKanjiCoverage(jdictDb)` — builds character sets from all slug and notes fields, then maps the top-2500 freq-ordered kanji from `jdict.db` to a status of `headword`, `notes`, or `none`.
- `getKanjiPages(char)` — queries `wiki_words` for pages containing the character in slug or notes and returns `{ headwordPages, notesPages }`.

**`index.js`**
- `GET /api/wiki/kanji-coverage` — returns the 2500-item coverage array.
- `GET /api/wiki/kanji/:char` — returns kanji data from KANJIDIC2 plus the two page lists.
- Frontend route `GET /wiki/kanji/:char` added.

## 2. Import image from URL

Added an **Import from URL…** button to the image actions row on wiki word pages (edit mode only).

Clicking the button toggles a URL input + Import button inline below the action row. Pressing Enter or clicking Import POSTs to the new backend endpoint.

### Backend — `POST /api/wiki/image-from-url`

- Accepts `{ url }` as JSON.
- **Google Images unwrapping** — if the URL is a `google.com/imgres?imgurl=…` link (what Safari's "Copy Link" produces), the real image URL is extracted from the `imgurl` query parameter before fetching.
- Sends browser-like `User-Agent` / `Accept` headers to avoid hotlink blocking.
- Follows up to 8 redirects.
- Streams the response to `wiki/images/<uuid>.<ext>` and returns `{ filename }`.

### Known limitation

The "Paste from clipboard" button is hidden on HTTP origins (local network access) because `navigator.clipboard.read()` requires a secure context (HTTPS or localhost). This is why it was invisible on iOS.

## 3. iOS image import investigation

Explored options for importing images from Safari on iOS without going through the photo library (which leaves unwanted clutter).

### What was tried / found

- **Web Share Target** (PWA manifest) — already implemented in a previous session; never worked reliably; reinstalling the PWA repeatedly made no difference. Not pursued further.
- **Clipboard paste** — requires HTTPS; not available on local HTTP.
- **URL import** — works if the user opens the image in a new tab and copies the address bar URL. Viable but requires several taps.

### Proposed solution — iOS Shortcut (to implement later)

Create a Shortcut that acts as a share extension for images:

1. **Receive** `Images` from Share Sheet
2. **Get Contents of URL** — POST to `http://SERVER:8000/api/wiki/image-upload`, body: Form, field `image` = Shortcut Input
3. **Get Dictionary from** response
4. **Get Dictionary Value** — key `filename`
5. **Open URLs** — `http://SERVER:8000/?pendingImage=[filename]`

The existing client-side `?pendingImage=` handler reads `sessionStorage('sharedImageTarget')` (set by the **Find on Google Images** button) and auto-navigates to the correct word page, applying and saving the image.

**Pre-condition:** tap **Find on Google Images** on the word page before sharing the image, so the target slug is stored in sessionStorage.
