# Development Report (10/03/2026)

## What this is

jdict is a locally-hosted Japanese dictionary web service backed by JMdict (~213k entries). It has two modes of use:

1. **Standalone** — browsing, searching, and building up a personal wiki of looked-up words
2. **Overlay** — called from a companion podcast PWA when the user taps a word in a synced transcript

The project was originally written in Python/FastAPI (now archived in `attic/`). It was migrated to Node.js/Express and then partially developed by Gemini before being rescued and rewritten in this session.

---

## Current architecture

### Backend (`index.js`, `db.js`, `wiki.js`)

These are solid and should not need significant changes.

- **`index.js`** — Express server. Serves the SPA for all frontend routes, and a REST API under `/api/`.
- **`db.js`** — SQLite access via `better-sqlite3`. Lazy singleton connection. Contains Japanese (kana/kanji LIKE + UNION) and English (FTS5) search logic.
- **`wiki.js`** — File-based wiki. Word and tag pages stored as markdown files with YAML frontmatter in `wiki/words/` and `wiki/tags/`. Images in `wiki/images/`. Uses `gray-matter` for parsing.
- **`importer.js`** — One-time JMdict XML → SQLite converter. Don't touch.

### Database schema

Three tables in `data/jdict.db`:
- `entries(seq, kanji_json, kana_json, senses_json, jlpt)` — main store
- `entries_text(seq, kanji, kana, glosses)` — denormalised for search
- `entries_fts` — FTS5 virtual table over glosses for English full-text search

### Frontend (`static/index.html`)

Single self-contained HTML file, no build step. Client-side router reads `window.location.pathname`. Views:

| Route | View |
|---|---|
| `/` | Search home |
| `/search/:word` | Search results pre-filled |
| `/entry/:seq` | Dictionary entry detail |
| `/wiki` | Wiki index (recently added/edited words and tags) |
| `/wiki/word/:slug` | Word wiki page |
| `/wiki/tag/:name` | Tag wiki page |

### Wiki page format

Markdown file with YAML frontmatter. Example `wiki/words/食べる.md`:

```markdown
---
seq: 1358280
word: 食べる
tags:
  - verbs
  - food
image: a3f2bc91-....jpg
contexts:
  - source: https://podcast.app/ep42?t=204
    timestamp: 2026-03-10T14:23:00.000Z
---
1. to eat
2. to live on (e.g. a salary)
```

Key fields:
- `seq` — JMdict sequence number, used to reliably fetch the dictionary entry
- `word` — display name (primary kanji form)
- `contexts` — lookup history; each entry has a `source` and `timestamp`

## The podcast app integration

### The concept

The companion app is a PWA that plays Japanese comprehensible input podcasts and displays a synced transcript. When the user taps a word in the transcript, jdict opens (probably as a bottom sheet / modal overlay) and the user can look the word up and optionally save it to the wiki.

### The URL contract

The podcast app opens jdict at:
```
/search/:word?source=<url>
```

- `:word` is whatever text was captured from the transcript tap. **Important:** due to HTML parsing of the transcript, this is often a fragment of a longer compound word, not the complete word. This is intentional — the search results list lets the user scan and select the correct full entry.
- `source` is a URL (or URL-like string) pointing back to the exact timestamp in the podcast where the word appeared. This gets stored in the wiki page's context list as a tappable backlink.

Example:
```
/search/食べ?source=https://podcast.app/episodes/42?t=204
```

### What jdict does with this

1. Opens with search results for the fragment
2. User finds the correct entry, taps it → entry detail page
3. If they tap "Add to Wiki", the `source` parameter is captured from the original URL and stored in the wiki page's context
4. The wiki page's lookup history then shows a tappable link back to that podcast moment

This is already implemented. The `pageSource` variable is read from `window.location.search` on page load and survives through to the `addToWiki` call.

### What is NOT yet implemented

- **Overlay mode** — when called from the podcast app, jdict should render without the search header and with a close/back button so the user can dismiss it cleanly. Suggest implementing as `?mode=overlay` query parameter. The podcast app would open `jdict` in a bottom sheet webview and pass this flag.
- **Auto-redirect if wiki page exists** — currently `/search/:word` always shows search results. You may want an option where, if a wiki page already exists for an exact match, jdict goes straight to it. Decided against this in this session because the word fragment problem means you always want to see results first. Revisit once you know how reliable the transcript tap is.

## Things already built but with no UI yet

These API endpoints exist and work but aren't exposed in the frontend:

- `GET /api/jlpt/:level` — returns entries tagged N1–N5. Note: JLPT data is not in JMdict itself; the `jlpt` column in the database needs to be populated separately from a JLPT word list.
- `GET /api/random?n=5` — returns random entries. Could become a discovery / flashcard feature.

## Suggested next steps for the integration session

1. **Define the source URL format** for the podcast app. Decide whether it will be a full `https://` URL, a custom scheme (`podcastapp://`), or a structured string. This determines how the backlink is rendered and whether it's directly tappable.

2. **Implement overlay mode** (`?mode=overlay`): hide the search header, add a prominent ← back button or close gesture, perhaps a slightly different background colour so it reads as a sheet. Small CSS/JS change.

3. **Test the word fragment problem** in practice. Once the podcast app is sending real taps, you'll know whether the fragments are close enough for search to find the right entry, or whether you need smarter matching (e.g. searching for any entry whose kanji *starts with* the fragment, which the current LIKE prefix query already does).

4. **Decide on wiki search.** Two options discussed:
   - Toggle in the header (Dictionary / Wiki mode) — more convenient once the wiki is large
   - Search box scoped to the wiki index page — simpler, keeps the two spaces separate
   Leaning towards the toggle, deferred until the wiki has enough pages to make it useful.

5. **Consider JLPT tagging** if structured study is a goal alongside the podcast listening.

## Running the service

```bash
npm install          # first time only
npm start            # production, port 8000
npm run dev          # development with nodemon auto-reload
```

The database (`data/jdict.db`) is already populated. Do not run the importer again unless rebuilding from scratch.

The wiki files live in `wiki/words/`, `wiki/tags/`, `wiki/images/` — these are the user's data and should be backed up separately (they are gitignored).
