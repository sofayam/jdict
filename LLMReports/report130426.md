# Session report — 2026-04-13

## 1. Wiki DB migration

Moved the file-based wiki (≈2,000 word pages, 277 card pages, 10 tag pages) from
`wiki/words/*.md` / `wiki/cards/*.md` / `wiki/tags/*.md` into a dedicated SQLite
database at `data/wiki.db`. Images remain on disk under `wiki/images/` served
statically as before.

### Schema

```sql
wiki_words  (slug PK, seq, tags JSON, image, contexts JSON, notes, created_at, updated_at)
wiki_cards  (slug PK, english, japanese, reading, image, notes, created_at)
wiki_tags   (name PK, notes, created_at, updated_at)
```

### Files changed

- **`wiki.js`** — complete rewrite. All functions are now synchronous (better-sqlite3).
  Same public API as before; `index.js` and the frontend required no changes.
  Added `saveCardPage` / `cardExists` (were missing; only ankiimport used them and
  that script is retired).
- **`wiki-migrate.js`** — new one-time migration script. Reads all `.md` files,
  preserves file `birthtime`/`mtime` as `created_at`/`updated_at`, bulk-inserts
  inside transactions. Run with `node wiki-migrate.js`; refuses to overwrite an
  existing `wiki.db`. After verifying, delete the old files:
  `rm -r wiki/words wiki/cards wiki/tags`.

### Why

The old code read every `.md` file on each request to `GET /api/wiki/browse` and
`GET /api/wiki/kana-index` — O(n) `fs.readFile` + gray-matter parse for every call,
with no caching. With ≈2,000 files this was the main cause of startup/navigation lag.
SQLite queries replace all of that with single indexed round-trips.

### Notes

- `getWikiIndexData` returns `mtime`/`birthtime` as ISO strings (from `datetime('now')`)
  rather than JS Date objects. JSON serialisation is identical; the frontend is
  unaffected.
- `getKanaIndex` / `getKanaWords` use a large `IN (?,?,…)` clause against `jdict.db`
  for the ≈2,000 wiki word seqs. Fine with modern SQLite (limit is 32,766); worth
  knowing if the wiki grows very large.
- `ankiimport.js` was not updated (retired).

## 2. Katakana search bug fix

### Symptom

Searching with a katakana form (e.g. コタツ) failed to find dictionary entries where
that katakana form existed but was not stored first in the space-separated kanji
column. Searching the hiragana equivalent (こたつ) worked because the hiragana form
happened to be stored first.

Example entry in `entries_text`:

```
kanji = "こたつ布団 炬燵布団 コタツ布団"
kana  = "こたつぶとん"
```

- `kanji LIKE 'こたつ%'` → TRUE (first token matches)
- `kanji LIKE 'コタツ%'` → FALSE (third token, not at column start)

### Root cause (`db.js`)

The Japanese-script WHERE clause used a column-prefix pattern:

```javascript
const prefixPattern = `${q}%`;
WHERE (t.kanji LIKE ?) OR (t.kana LIKE ?)
```

This matches only if `q` is at the very start of the entire column value. When JMdict
stores multiple written forms space-separated, non-first forms are invisible to this
search.

### Fix

Changed to a word-boundary prefix pattern applied to the padded column:

```javascript
const searchPattern = `% ${q}%`;
WHERE (' ' || t.kanji || ' ' LIKE ?) OR (' ' || t.kana || ' ' LIKE ?)
```

Prepending/appending a space means `% q%` anchors to any word boundary within the
column, not just the column start.

### Tradeoff

The leading `%` prevents SQLite from using the B-tree indexes on
`entries_text(kanji)` and `entries_text(kana)`, causing a full table scan (215k rows).
For a local single-user app this is fast enough; worth revisiting if it ever becomes
noticeable (options: store forms in separate rows at import time, or add a dedicated
search index).
